# Package inventory

## Overview

Map of source repos (under `/Users/spencer/workspaces/spencerbeggs/`) to target `@effected/*` packages. **The migration program is complete** (the final package, `@effected/app`, merged 2026-07-12); this doc is the permanent record of where each source repo went and what each migration established. Post-`0.1.0` work lives in [roadmap.md](roadmap.md). Tier definitions are in [effect-standards.md](effect-standards.md). Tiers classify libraries: the one published package here that is not a library, `@effected/pnpm-plugin-effect`, carries no tier and is a [companion](effect-standards.md#companion-packages-published-but-not-a-library) instead.

## Cross-package realignment (2026-07-08)

`chore/realignment` is a cross-package cleanup pass over the five landed packages (semver, jsonc, yaml, package-json, npm), not a new migration. Three shifts touch package state:

- **Inline API-Extractor factories everywhere.** All five packages were converted from the transitional `@public X_base` idiom to the inline class-factory form with a narrow `_base` suppression in each `savvy.build.ts`, per the [effect-standards API-Extractor policy](effect-standards.md#api-extractor--effect-class-factories). The `@public X_base` backlog is fully cleared; every package retains a zero-warning `issues.json`.
- **yaml input-hardening extended.** Beyond the composer/CST depth caps already recorded in [packages/yaml.md](packages/yaml.md), two more defect surfaces now fail typed: an alias-expansion "billion laughs" bomb that stayed under `maxAliasCount` but OOM-crashed the heap (now bounded by a materialized-node budget, fatal `AliasCountExceeded`), and deep-input stack overflows in `Yaml.stringify` (value path) and `YamlDocument.stringify` (node path), now capped at `MAX_NESTING_DEPTH = 256` (fatal `NestingDepthExceeded`).
- **Structured error shapes.** jsonc's `JsoncModificationError` moved from a `reason: string` to typed `expected`/`depth` fields (the structure-preserving-errors house rule; yaml's `YamlModificationError` was already compliant), and package-json's `ScopedPackageName`/`UnscopedPackageName`/`SpdxLicense` branded types now export as `string & Brand.Brand<…>` rather than `typeof X.Type`.

## Migration order

Dependency sequencing came from the review synthesis (`.claude/reviews/SYNTHESIS.md`), rescoped 2026-07-09 by the [release gate](releases.md#the-gate). The sequence is complete.

All **seventeen library packages are merged**: semver — jsonc — yaml — package-json (+ npm) — config-file — walker — glob — toml — lockfiles (#9) — store (#10) — xdg (#11) — runtimes (+ `-cli`) (#12) — workspaces (#13) — ts-vfs (#14) — app (#15, merged 2026-07-12 via PR #73). Along the way [the config-file consolidation](#the-config-file-consolidation-2026-07-11) deleted the three adapter packages, which is why fifteen migrations produced seventeen packages rather than twenty.

**app** was sequenced last, on purpose ([packages/app.md](packages/app.md)). It is a thin composition layer over `xdg` + `config-file` + `store` with no domain logic of its own, so it is the one package whose content is decided entirely by how its consumers actually wire the kit together. Sequencing it last let it absorb the wiring that the ts-vfs port exercised for real, rather than guessing at a composition and then discovering the guess was wrong — and there was a concrete specimen to absorb: the layer stack ts-vfs deliberately does **not** build for its consumers (platform layers + store's `Cache` + `TypeCache.layerXdg` + `PackageFetcher.layer` + `TypeRegistry.layer`), recorded in [packages/ts-vfs.md](packages/ts-vfs.md). It lands in the design as a **documentation specimen only** — `app` takes no `ts-vfs` edge — together with that port's three wiring findings. **No consumer is blocked on it**, because nothing may depend on it — a library taking an application control plane as a dependency would be an [R2](effect-standards.md#dependency-policy) tier-3 leak — so deferring it blocked nothing.

The consolidation ran first, ahead of both ports, because it changed the shape of the kit rather than its contents: doing it then meant the two remaining ports, the release gate and every consumer's install instructions were written **once** against the final package set instead of against the family and then rewritten. It also shrank what `0.1.0` has to version and publish, from twenty library packages to seventeen.

`json-schema` is off the roadmap entirely.

### The config-file consolidation (2026-07-11)

**Executed 2026-07-11 on `feat/remerge-config-file`. Supersedes [the config-file family](#the-config-file-family) below, which records why the family was created and is kept for that history.**

`@effected/config-file-jsonc`, `@effected/config-file-yaml` and `@effected/config-file-toml` are **deleted**; `@effected/config-file` absorbed their three codecs. The three **format** packages — `@effected/jsonc`, `@effected/yaml`, `@effected/toml` — remain independent packages and were untouched; only the adapter shims went. **The workspace went from 19 packages to 16, and the merged count from eighteen to fifteen.**

The family's original justification was a tooling fact: this monorepo does not use subpath exports, so every optional dependency became a package boundary. That framing made the split read as a workaround, and it is not the reason the decision rested on. The evidence was:

- **There is no dispatch table.** `ConfigCodec` is an interface and each codec is a single **named** export — `JsonCodec`, `JsoncCodec`, `YamlCodec`, `TomlCodec` — which the caller passes explicitly to `ConfigFile.layer` as `options.codec`. Nothing in the pipeline maps a file extension to a codec; `ConfigCodec.extensions` was pruned at port time precisely because nothing read it.
- **Because codecs are explicitly composed, a bundling consumer tree-shakes every codec it does not reference**, and the unreferenced engine's whole module graph goes with it. Asserted when the decision was taken and **since measured**: a `JsonCodec`-only consumer bundles 506 bytes of the merged package; a `TomlCodec`-only consumer 26.5 kB with the TOML engine and nothing else; all four codecs 129.4 kB. See [packages/config-file.md](packages/config-file.md#as-built-the-tree-shaking-property-is-measured-not-assumed).
- **A non-bundling Node consumer pays nothing either.** ESM never loads a module nobody imports, so a TOML-only consumer never executes the yaml engine even unbundled.
- **pnpm's content-addressed store hardlinks**, so the install cost of the extra dependency edges is negligible. Install weight was the original argument for splitting the codecs out, and it does not survive contact with how pnpm actually works.
- **The DX gain is the point**: one install, `TomlCodec` just works, and three fewer packages to version, release and document.

As-built, the load-bearing constraint was honored in the **stronger** of the two ways it permitted: rather than keeping a JSON-only `ConfigCodec` namespace object and adding the format codecs beside it, the object was **deleted outright**. `ConfigCodec` is now the interface only; the four codecs are free-standing named exports, one module each, and `ConfigCodec.json` became `JsonCodec` (a breaking rename, free because nothing is published). With no object to grow, "never a dispatch table" is structurally impossible to violate rather than a convention someone could later tidy away. `@effected/xdg` was the only code consumer of the old name.

The honest costs — chiefly that config-file's zero-runtime-dependency property is spent, now accurately "zero **external** runtime dependencies" — are recorded in [packages/config-file.md](packages/config-file.md#the-consolidation-2026-07-11). Every gate stayed green through the merge, with `issues.json` unchanged.

### The ts-vfs rename

**Approved 2026-07-11.** The `type-registry-effect` migration target is `@effected/ts-vfs`, not `@effected/type-registry`.

The package fetches, caches and resolves TypeScript type definitions from npm packages so that Twoslash and similar documentation tooling can typecheck type-aware code samples; it wraps `@typescript/vfs`. Its v3 modules are `TypeRegistry`, `VirtualPackage`, `events`, `metrics` and `node`.

The reasoning, recorded so it is not re-litigated: `type-registry` describes the v3 implementation's central facade rather than what the package is for. A bare `vfs` would over-generalize in the other direction — it would signal a general-purpose in-memory filesystem, when this package makes network calls to npm and caches to a SQLite store. **`ts-vfs`** scopes the name to TypeScript, matches the `@typescript/vfs` package it wraps and does not overclaim. The rename changes nothing about the migration: it is still a target in its own right, still integrated tier, still load-bearing for two of the five gate applications.

The **v3 source repo and its published package keep the name `type-registry-effect`** — that name still appears in `rspress-plugin-api-extractor`'s dependency list, and that reference is about the old package, not the new one. (It also used to appear in a v3/v4 peer-poisoning note in the root `CLAUDE.md`: `type-registry-effect` peered on `effect`, which under the old pnpm resolver bound the website's transitive `@effect/platform` to a v4 core. That bug is fixed upstream in pnpm 11.12.0 and the note is gone.)

### The app rename

**Approved 2026-07-12.** The remaining composition package is **`@effected/app`**, not `@effected/app-kit`.

The reasoning, recorded so it is not re-litigated: `app-kit` collided with the repo's established vocabulary, where "the kit" names the whole seventeen-package release — a package called `app-kit` reads as an umbrella over the kit, which is exactly what [its inventory entry](#internal-packages-no-source-repo) says it must not be. The package is an application control plane — the wiring layer an application composes at its edge — and `@effected/app` says that without overclaiming, the same test [the ts-vfs rename](#the-ts-vfs-rename) applied.

The rename happened before any code existed, so it cost a documentation pass only; after scaffolding it would have touched manifests, catalogs and the release-gate table.

### The config-file family

**Dissolved 2026-07-11 by [the config-file consolidation](#the-config-file-consolidation-2026-07-11) — the three adapters were deleted and their codecs moved into the core. This section is kept because it records how the family came to exist and what each member shipped; it is history, not the plan.**

Because this monorepo does not use subpath exports, every optional dependency of `@effected/config-file` became a package boundary. Migration #5 therefore delivered a family rather than a package, each member on its own spec → plan → implement cycle:

| Package | Tier | Order | Fate |
| --- | --- | --- | --- |
| `@effected/config-file` (core pipeline + JSON codec) | boundary | 5a | **merged**; absorbed the three codecs |
| `@effected/config-file-jsonc` | pure | 5b | shipped, then **deleted by the consolidation** — now `JsoncCodec` |
| `@effected/config-file-yaml` | pure | 5c | shipped, then **deleted by the consolidation** — now `YamlCodec` |
| `@effected/toml` | pure | 5d | **merged**; a *format* package, untouched ([packages/toml.md](packages/toml.md)) |
| `@effected/config-file-toml` | pure | 5e | shipped, then **deleted by the consolidation** — now `TomlCodec` |
| `@effected/config-file-watcher` | boundary | 5f | not started — **off the gate**; unaffected, still its own package |

5a–5c landed together on `feat/config-file`; the core never depended on `@effected/toml`, so the TOML port did not block migration #5. Dependency direction was and remains strictly acyclic: config-file → format packages, never the reverse. The consolidation moved the three codecs into the core without giving any format package knowledge of config-file, so it stays acyclic. 5d is on the [release gate](releases.md#the-gate) and 5e was — `@soda3js/config` is the TOML consumer that put them there, and that consumer's gate rows are now `config-file` plus `toml`, the adapter row having ceased to exist. 5f is not on the gate and needs redesign rather than translation; the consolidation did not touch it, because a watcher is a boundary-tier service rather than a codec and was never a candidate for absorption on tree-shaking grounds.

When `@effected/walker` landed, `config-file`'s `internal/walkUp.ts` was deleted and its `ConfigResolver` strategies re-expressed over walker's primitives. That was the first change to config-file's then-zero-dependency profile; the consolidation was the second and larger one. Nothing is published, so both cost a refactor commit rather than a breaking release.
