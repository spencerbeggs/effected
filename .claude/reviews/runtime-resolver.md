# Review: runtime-resolver → @effected/runtime-resolver

Source repo: `/Users/spencer/workspaces/spencerbeggs/runtime-resolver` (v0.3.21, Effect 3.x)
Reviewed against: `.claude/design/effected/effect-standards.md`
Verdict: **Effect v3 library, BOUNDARY tier confirmed.** Solid layered design worth
redesigning (not lifting) — the domain concepts are right; the file layout, per-runtime
duplication, and a few state/provenance decisions are not.

## 1. What it does / Effect status

Resolves semver-compatible versions of Node.js, Bun, and Deno runtimes. Despite the
name lacking an `-effect` suffix, it is **fully Effect v3-based** end to end: 8 `Context.Tag`
services, 23 layers, Effect Schema codecs, `Data.TaggedError` errors, an `@effect/cli`
binary, and a thin Promise facade (`resolveNode`/`resolveBun`/`resolveDeno`) that
provides prebuilt default layers and `Effect.runPromise`s for non-Effect consumers.

Data sources: `nodejs.org/dist/index.json` (Node versions), the `nodejs/Release`
schedule (LTS phases), GitHub tags/releases for Bun (`oven-sh/bun`) and Deno
(`denoland/deno`) via Octokit. Three cache strategies per runtime:

- **Auto** — fetch live, silently fall back to bundled snapshot on network/parse failure
- **Fresh** — live data required; fails with `FreshnessError` otherwise
- **Offline** — bundled snapshot only, no IO

Bundled snapshots (`src/data/*-defaults.ts`, ~6.8k generated lines) are regenerated by
a build script, giving genuine offline/CI resilience. It depends on `semver-effect`
(`SemVer`, `Range`, `VersionCache`) for all version math.

Layout is the kind-based one the standards explicitly supersede: `errors/`, `layers/`,
`schemas/`, `services/`, `lib/`, `data/`, `cli/` — 23 files in `layers/` alone.

## 2. What is done well (preserve these)

- **Cache-strategy-as-layer.** Auto/Fresh/Offline are swappable layers over one cache
  service interface. This is the package's signature DX idea and the redesign should
  keep it first-class (as named layer exports on each resolver's service class).
- **Clean vertical dependency graph.** `Resolver → ReleaseCache → Fetcher(s) →
  GitHubClient → OctokitInstance ← auth layers`. Transport, auth, fetching, caching,
  and resolution are genuinely separated; each seam is independently mockable and the
  tests exploit that.
- **`OctokitInstance` as a narrow structural port.** The service interface declares only
  the two Octokit methods actually used (`repos.listTags`, `repos.listReleases`), so
  tests stub a tiny object instead of the whole Octokit type. Textbook ports-and-adapters.
- **Typed HTTP error ladder.** `mapOctokitError` maps status codes to
  `AuthenticationError` (401), `RateLimitError` (403/429, with `retryAfter`/limit
  headers parsed), `NetworkError` (rest); `retryOnRateLimit` composes
  `Schedule.exponential("1 second")` + `Schedule.recurs(3)` gated by
  `Predicate.isTagged("RateLimitError")`. All of this translates directly to v4.
- **Auth layer triad.** `GitHubTokenAuth` / `GitHubAppAuth` / `GitHubAutoAuth`
  (env-priority detection: App > PAT > `GITHUB_TOKEN` > unauthenticated, with a
  warning on ambiguity). The precedence policy and its documentation are good design.
- **Generic `RuntimeCache<R extends RuntimeRelease>`** built over semver-effect's
  `VersionCache` with a version→release lookup map — one implementation serves all
  three runtimes; `Effect.die` on lookup-map inconsistency is a correct defect (invariant
  violation, not a business failure).
- **Node lifecycle model.** `NodeSchedule.phaseFor(major, now)` with an explicit
  reference-date parameter makes phase logic (`current`/`active-lts`/`maintenance-lts`/
  `end-of-life`) deterministic and testable without clock mocking.
- **Documentation discipline.** Every export has TSDoc with `@public`/`@internal`,
  worked examples, failure-mode lists, and cross-links; `docs/` covers the Promise API,
  Effect API, auth, offline fallback, and CLI separately. Keep this bar.
- **CLI ergonomics.** Structured JSON output with a `$schema` reference and a
  `--schema` flag that prints the published JSON Schema.

## 3. What is confusing or awkward

These are design issues, not v3-idiom nitpicks:

- **The `source` provenance feature is broken.** README advertises `source: "api" |
  "cache"` as a headline feature, but all three resolvers hardcode `source: "api"`.
  The Auto cache layer knows whether it fell back, but that knowledge dies at the layer
  boundary — nothing ever produces `"cache"`. The redesign must make provenance part
  of the cache service's state.
- **3× mechanical duplication.** Auto/Fresh/Offline × {Node, Bun, Deno} + three
  resolvers + four fetchers + three release caches = ~21 near-identical layer files.
  The Fresh layers for Bun and Deno differ by literally a repo name. This is the
  combinatorial explosion the module-per-concept layout exists to kill; the generic
  machinery belongs in `src/internal/`, parameterized once.
- **`NodeRelease` carries a `Ref<NodeSchedule>` inside every domain value.** Each
  release instance holds a shared mutable reference so `release.phase(now)` can consult
  the schedule. Mutable service state threaded through immutable domain values (the
  file itself apologizes: it can't be a `Data.TaggedClass` because the Ref breaks
  equality). Phase should be a function of `(release, schedule, now)` — schedule owned
  by the cache/resolver, not the model.
- **`createRuntimeCache` uses a bare closure `Map` with a documented "load is not
  concurrency-safe" caveat.** A comment is not a concurrency strategy; this is
  `Ref`/`SynchronizedRef` territory.
- **`GitHubClient` is misnamed and boundary-blurred.** Its `getJson` uses raw global
  `fetch` (not a platform HttpClient) with an ad-hoc `{ decode }` pseudo-schema
  parameter, and `NodeVersionFetcherLive` fetches *nodejs.org* through the
  "GitHub" client. It's really an HTTP client with two Octokit conveniences bolted on.
- **`GitHubAutoAuth` reads `process.env` directly** at layer construction (no
  `Config`), and constructs-then-provides another layer *inside* a layer
  (`Effect.provide(OctokitInstance, appLayer)`) to delegate to App auth.
- **Silent error collapse in resolvers.** `cache.filter(range)` failures are
  `catchAll`-ed into an empty array, so an *invalid* semver range surfaces to the user
  as `VersionNotFoundError` ("no versions found") instead of `InvalidRangeError`.
  Standards violation in spirit: never collapse distinct failures early.
- **Dead exports.** `InvalidInputError` and `CacheError` are exported from `index.ts`
  but never constructed anywhere in `src/`.
- **Redundant `message: string` on every error** duplicating what the structured
  fields (`runtime`, `constraint`, `url`, `status`, ...) already encode.
- **`Layer.tap((ctx) => setup.pipe(Effect.provide(ctx)))`** as the mechanism for
  populating caches at construction — clever, opaque, and unnecessary once cache
  construction is a single `Layer.effect`.
- **Wall-clock defaults via `DateTime.unsafeMake(new Date())`** scattered through
  resolver and model code instead of `Clock`-derived time.
- **No observability at all.** Zero `Effect.fn`, zero spans/annotations; one
  `logWarning` in the entire package.
- **Tests are plain vitest + `Effect.runPromise` + repeated `Effect.provide` per test**
  — exactly the anti-pattern the testing standard names. Coverage is broad
  (per-layer/per-service tests exist), so migration to `it.effect` + top-level
  `layer(...)` groups is mechanical, not a rewrite.
- **Options are plain interfaces, not schemas**, so the CLI re-validates
  phases/increments by hand instead of decoding once.

## 4. v4 migration implications

This is a redesign, per the playbook. Mapping:

| Current (v3) | Target (v4) |
| --- | --- |
| `Context.Tag` classes + separate `*Live.ts` layer files | `Context.Service` classes with layers co-located in the same concept file |
| `Data.TaggedError` × 8 (2 dead) | `Schema.TaggedErrorClass` ladder; wrapped Octokit/fetch failures get `cause: Schema.Defect`; drop free-text `message` fields; delete `InvalidInputError`/`CacheError` |
| `Schema.Struct` + type alias pairs (`ResolvedVersions`, etc.) | `Schema.Class` models with static/instance methods; construct via `.make` |
| Plain-class `NodeRelease` + `Ref<NodeSchedule>` + `fromInput` statics + parallel `*Input` structs | `Schema.Class` with field-level `decodeTo` transformations (raw version strings → `@effected/semver` SemVer schema); phase becomes `release.phase(schedule, now)` — no Ref in the model |
| `Schema.optional` for `lts`/`default` | `Schema.optionalKey` |
| Raw `fetch` in `getJson`; `@effect/platform` peer (never imported) | `HttpClient` from `effect` core (v4 merged platform); consumer provides platform layer at the edge |
| `process.env` reads in auth layers | `Config`/`ConfigProvider` (redemption: keeps the same precedence policy, testable) |
| `DateTime.unsafeMake(new Date())` defaults | `DateTime.now` / `Clock` in the effect context |
| Anonymous `Effect.gen` everywhere | `Effect.fn("NodeResolver.resolve")` etc. at operation boundaries; `annotateCurrentSpan({ runtime, range })` |
| plain vitest + `runPromise` | `@effect/vitest` `it.effect`, top-level `layer(...)` groups, tests in `__test__/` |
| `semver-effect` regular dep | `@effected/semver` `workspace:*` (its own v4 redesign is a prerequisite — `RuntimeCache` is built on its `VersionCache`, `Range`, `SemVer`) |

Design decisions to make at design-doc time:

1. **Drop Octokit?** The library uses exactly two trivial REST GETs
   (`/repos/{o}/{r}/tags`, `/repos/{o}/{r}/releases`). Reimplementing them on
   `HttpClient` removes `octokit` + `@octokit/auth-app` (the package's entire regular
   dependency weight) and the `OctokitInstance` port with it. The one hard part is
   GitHub App auth (JWT signing + installation-token exchange); options: keep
   `@octokit/auth-app` as an *optional* peer behind a separate App-auth layer, or defer
   App auth to a future `@effected/github` package. Recommended: drop Octokit, keep
   token/auto auth in-package, make App auth pluggable.
2. **Split the CLI out.** `bin` forces `@effect/cli` + `@effect/platform-node` into the
   library's peer closure for every API consumer. Either a sibling tool package
   (CLI = app tier, full regular deps) or a `./cli` subpath export with optional peers.
   Recommended: separate package; the library stays services-and-layers only.
3. **Fix provenance.** The cache service tracks `source` (set at load time by whichever
   strategy populated it, held in the same `Ref` state as the releases); resolvers read
   it instead of hardcoding `"api"`. The Auto strategy's silent fallback should also
   `logWarning` — silently serving stale data with `source: "api"` is the current
   worst case.
4. **Surface `InvalidRangeError`.** Resolver error channel becomes
   `InvalidRangeError | VersionNotFoundError` (range problems are the caller's bug to
   see, not a "not found").
5. **Collapse the strategy × runtime matrix.** One internal parameterized
   cache/fetch/resolve core; public per-runtime concept files expose
   `layers.auto` / `layers.fresh` / `layers.offline` constants (bound to constants for
   layer memoization per standards).

## 5. Candidate module-per-concept layout

~15k source lines shrink substantially: ~6.8k is generated snapshot data (moves to
`internal/defaults/`), and the 21-file layer matrix collapses into the concept files.

```text
src/
  index.ts                 # re-exports only
  ResolvedVersions.ts      # Schema.Class result model; Source + Increments literals;
                           #   VersionNotFoundError (raised by all resolvers)
  NodeRelease.ts           # Schema.Class (version/npm/date via decodeTo) + NodePhase
  NodeSchedule.ts          # Schema.Class schedule model + phaseFor(major, now)
  NodeResolver.ts          # Context.Service; resolve(options); options schema;
                           #   layers: NodeResolver.Auto / .Fresh / .Offline
  BunRelease.ts            # Schema.Class
  BunResolver.ts           # Context.Service + Auto/Fresh/Offline layers
  DenoRelease.ts           # Schema.Class
  DenoResolver.ts          # Context.Service + Auto/Fresh/Offline layers
  GitHub.ts                # GitHub API client Context.Service (HttpClient-based);
                           #   auth layers (Token / Auto / App-pluggable) + Config-based
                           #   env detection; AuthenticationError, RateLimitError,
                           #   NetworkError; FreshnessError if kept
  internal/
    releaseIndex.ts        # generic Ref-based cache over @effected/semver VersionCache
                           #   (replaces RuntimeCache/createRuntimeCache); provenance state
    fetch.ts               # parameterized fetch pipelines (dist index, tags, releases),
                           #   retryOnRateLimit, tryParseSemVer
    strategy.ts            # auto/fresh/offline construction shared by the 3 resolvers
    defaults/
      node.ts bun.ts deno.ts   # generated snapshots (regeneration script kept)
__test__/
  NodeResolver.test.ts  BunResolver.test.ts  DenoResolver.test.ts
  GitHub.test.ts  NodeSchedule.test.ts  ReleaseIndex.test.ts
  integration/  e2e/
```

Open question for the design doc: whether `*Release.ts` files fold into their
`*Resolver.ts` files (release model + resolver service are arguably one concept per
runtime; Node's schedule complexity argues for keeping Node's split, Bun/Deno could
merge). Promise facade (`resolveNode` etc.): keep as statics on the service classes
(`NodeResolver.resolvePromise(options)`) or drop entirely in the Effect-first
monorepo — decide at design time; the prebuilt default-layer composition it embodies
should survive as an exported `NodeResolver.Default` layer either way.

## 6. Dependency hygiene and tier

**Tier: BOUNDARY — confirmed** (network IO to GitHub/nodejs.org, env/config access).
Programs against `HttpClient` from `effect` core; consumer provides
`@effect/platform-node`/bun at the edge. The bundled-defaults Offline layer is pure,
but the package's purpose is IO.

Current manifest problems:

- **Phantom peer:** `@effect/platform` is declared as a peer but never imported
  anywhere in `src/`.
- **CLI peers leak onto library consumers:** `@effect/cli` and `@effect/platform-node`
  are peers of the whole package but are used only by the `bin` entry. Every API-only
  consumer inherits them (README even tells users auto-install will pull them).
- **Heavy regular deps:** `octokit` + `@octokit/auth-app` — large transitive graphs
  for two GET endpoints (see decision 1 above).
- **`semver-effect` as a versioned regular dep** → becomes `@effected/semver`
  `workspace:*`; sequencing constraint: @effected/semver's v4 port lands first.

Target manifest (library, post-split, post-Octokit-removal):

- `peerDependencies`: `effect` (v4) — complete closure, nothing transitive left
  undeclared (standards §peer-dependency discipline; systems#228 lesson).
- `dependencies`: `@effected/semver` (`workspace:*`; peer-vs-regular decided per edge
  at design time).
- Optional peer only if App auth stays: `@octokit/auth-app` (marked optional, with the
  App-auth layer in its own module so bundlers tree-shake it).
- CLI package (separate): regular deps on `effect`, `@effect/cli`,
  `@effect/platform-node`, `@effected/runtime-resolver` (app tier: full stack as
  regular dependencies per standards).
