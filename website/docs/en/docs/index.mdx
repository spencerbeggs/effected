# effected

A pnpm monorepo (npm org `@effected`) building an [Effect](https://effect.website/) v4 app kit: a set of
libraries designed for Effect v4 from the start, not lifted from their v3 predecessors. The repo holds
libraries only — the applications that consume them stay in their own repos.

Scope is closed by five consuming applications, not by how many source repos remain. A library earns its
place by being something one of those five actually needs.

## Nothing is published yet

Nothing here is on npm. The whole kit ships together at `0.1.0`, once it can replace the business logic of
the five applications that define its scope, and `1.0.0` waits for Effect v4 GA. That includes
[@effected/pnpm-plugin-effect](/pnpm-plugin-effect): it is not a library, so the tier taxonomy does not
describe it, but it is a public package and it ships in the same release.

No package is released on its own. That is what buys one internally consistent graph on day one: every
package's peer range names the same Effect v4 beta, and no consumer ever installs a half-built kit. Until
then the packages are developed against the beta pinned in the `effect` catalog, and a beta bump is a
coordinated change across the whole kit rather than a per-package decision.

The API reference in these pages therefore describes packages you cannot yet install. It is generated from
the source that will ship.

## Tiers

Every **library** declares a tier, which describes its runtime surface:

* **pure** — peers on `effect` and takes only `@effected/*` edges, with no IO.
* **boundary** — the same dependency surface, but does IO through Effect's core `FileSystem` and `Path`
  services.
* **integrated** — imports at least one runtime package outside `effect` core.

The tier is a dependency statement rather than an IO one. A package with no IO at all still vendors its
engine rather than take a runtime dependency, which is why `@effected/glob` is pure despite carrying a
ported matcher. A library's tier tells you what depending on it will cost you.

### Companion packages

A **companion** package is published and installable but is **not a library**: it exposes no API, so there
is nothing to import and nothing to call. Companion is a category, not a fourth tier — the three tiers sit
on one axis, dependency surface, and that axis is meaningless for a package nothing can depend on. A
companion has no tier rather than a fourth one.

[@effected/pnpm-plugin-effect](/pnpm-plugin-effect) is the only companion today. It ships two pnpm catalogs
and a pnpmfile — configuration, not code. Installing it is optional for you; shipping it is not optional for
the release, and it goes out at `0.1.0` with the libraries.

## Packages

| Package | Tier | Description |
| ------- | ---- | ----------- |
| [@effected/semver](/semver) | pure | Strict SemVer 2.0.0 versions, ranges and comparators as Effect schemas |
| [@effected/jsonc](/jsonc) | pure | Zero-dependency JSONC parsing, editing and formatting as Effect schemas |
| [@effected/yaml](/yaml) | pure | Zero-dependency YAML parsing, editing and formatting as Effect schemas |
| [@effected/toml](/toml) | pure | TOML 1.0.0 parsing, editing and formatting as Effect schemas, with typed diagnostics, a lossless CST and first-class date-time values |
| [@effected/package-json](/package-json) | integrated | package.json parsing, editing, validation and file IO as Effect schemas |
| [@effected/npm](/npm) | pure | Effect service contracts for resolving pnpm `catalog:` and `workspace:` dependency specifiers |
| [@effected/config-file](/config-file) | boundary | Composable config file loading: JSON, JSONC, YAML and TOML codecs, resolution strategies and merge behaviors |
| [@effected/walker](/walker) | boundary | Upward path traversal as Effect primitives: ascend a directory chain and return the first candidate satisfying a predicate |
| [@effected/glob](/glob) | pure | Full-fidelity glob matching as Effect schemas: the complete minimatch dialect compiled to pure string predicates |
| [@effected/lockfiles](/lockfiles) | pure | Lockfile parsing for bun, npm, pnpm and yarn Berry into one unified Effect schema model, with pure integrity checking against workspace manifests |
| [@effected/store](/store) | integrated | Durable local state: a schema-versioned, migrated store, and a cache built on it |
| [@effected/xdg](/xdg) | boundary | XDG Base Directory resolution: app directories, native directories and config-file resolvers |
| [@effected/runtime-resolver](/runtime-resolver) | boundary | Resolve semver-compatible Node.js, Bun and Deno versions from the live release feeds, with a bundled offline snapshot |
| [@effected/runtime-resolver-cli](/runtime-resolver-cli) | integrated | The `runtime-resolver` binary, split out so the library's consumers never install a platform adapter |
| [@effected/workspaces](/workspaces) | integrated | Monorepo tooling: discovery, the dependency graph, package-manager detection, pnpm catalogs, lockfile IO and git change detection |
| [@effected/pnpm-plugin-effect](/pnpm-plugin-effect) | companion — no tier | The pnpm config dependency publishing the two Effect catalogs every package here pins against |

Installing [@effected/pnpm-plugin-effect](/pnpm-plugin-effect) holds your `effect` versions and peer floors
at exactly the values this kit was built and tested against, rather than leaving you to resolve your own.

## What is left

Two packages remain before the kit is complete, in order: `ts-vfs`, then `app-kit`. `ts-vfs` is the last
package with real domain logic to port — it fetches, caches and resolves TypeScript type definitions from
npm so Twoslash-style tooling can typecheck type-aware samples. `app-kit` goes last because it is a thin
composition over `xdg`, `config-file` and `store`, and what belongs in it is decided by how consumers
actually wire the kit.
