# @effected/runtime-resolver-cli

The `runtime-resolver` binary. Resolves Node.js, Bun and Deno versions through [`@effected/runtime-resolver`](/runtime-resolver) and prints a JSON envelope, so a CI job can pipe it into `jq` and pick out a version to install.

## Why a separate package

This package exists to be the one that carries `@effect/platform-node`.

A command-line tool needs a concrete runtime: a real filesystem, a real terminal, a real process. Effect core *declares* those services and implements none of them for Node — `@effect/platform-node` supplies the Node implementations. That is exactly the dependency a library's consumers must not be made to install, so the binary becomes its own package rather than making every API-only consumer of the resolver pay for a platform adapter.

The library stays clean; the tool declares its full stack. Nothing depends on this package, so nothing inherits its weight.

## Install

```bash
npm install --save-dev @effected/runtime-resolver-cli
```

```bash
pnpm add -D @effected/runtime-resolver-cli
```

Or run it without installing:

```bash
npx @effected/runtime-resolver-cli --node ">=20"
```

Requires Node.js >=24.11.0.

## Quick start

Ask for the Node versions matching a range and get a JSON envelope back:

```bash
runtime-resolver --node ">=20" --pretty
# {
#   "ok": true,
#   "results": {
#     "node": {
#       "ok": true,
#       "source": "api",
#       "versions": [ ... ],
#       "latest": "...",
#       "lts": "..."
#     }
#   }
# }
```

The envelope shape is the point: `results` is keyed by runtime, each entry carries `ok`, and a successful entry carries `source`, `versions` and `latest`. Pipe it into `jq` and a CI job has its version to install.

Resolve several runtimes at once, or pin the whole thing to the offline snapshot so the job performs no network IO:

```bash
runtime-resolver --node ">=20" --bun "^1.0.0" --deno "^2.0.0" --offline
# JSON envelope with a "node", "bun" and "deno" entry, each source: "cache"
```

`--schema` prints the JSON Schema of the response and exits, so a consumer can validate the envelope rather than trusting its shape.

## The flags

`--node`, `--bun` and `--deno` each take a semver range. `--node-phases` takes a comma-separated list of Node lifecycle phases to accept, and `--node-date` evaluates those phases at a given date instead of now. `--increments` chooses the granularity at which matching versions are grouped (`latest`, `minor` or `patch`). `--node-default`, `--bun-default` and `--deno-default` each take a range whose newest match becomes that runtime's `default` field.

`--offline` uses the bundled snapshot only. `--pretty` indents the output. `--token` supplies a GitHub token, overriding `GITHUB_PERSONAL_ACCESS_TOKEN` and `GITHUB_TOKEN`; Bun and Deno resolve through GitHub's API, while Node needs no credentials at all.

## A usage error exits 1, a resolution that matches nothing exits 0

The rule the tests exist to hold, and the one easiest to break by "improving" the output.

A **usage** error — no runtime selected, an unrecognized `--node-phases` value — is a failure. It exits non-zero. Printing the complaint and returning successfully exits `0`, and a CI job gating on the exit status then reads a typo in its own config as a pass.

A **resolution that matches nothing** is data. It exits `0` with `ok: false` in the envelope, because a caller asking which Node versions match `>=999` got a real, correct answer: none. Collapsing the two in either direction is the bug.
