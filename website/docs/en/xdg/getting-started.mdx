# Getting Started

Every command-line tool has to answer the same question: where does this user's config, data, cache and state belong on disk? On Linux the answer is the XDG Base Directory specification; on macOS and Windows it is a different convention entirely. `@effected/xdg` resolves all of it for one application namespace and hands you the directories as plain strings — with on-demand creation for the moment you actually need them on disk. Reach for it whenever a tool needs a home for a user's files and you want those paths correct on every OS without hand-rolling `~/.config` string math.

This guide assumes you have never used the package. If you are also new to Effect v4, the only idioms you need here are `Effect.gen(function* () { … })` with `yield*` to sequence steps, `Effect.provide` to supply a service, and `Effect.catchTag` to route a typed failure — all shown in full below.

## Install

```bash
npm install @effected/xdg effect @effect/platform-node
```

```bash
pnpm add @effected/xdg effect @effect/platform-node
```

Requires Node.js >=24.11.0. `effect` v4 is a peer dependency, and so are `@effected/walker` (the search primitive the config resolvers are built on) and `@effected/config-file` (whose resolver seam `XdgConfig` implements). Package managers that install peers automatically pull them in; add them to your manifest explicitly if yours does not. The package declares no runtime dependencies of its own. Creating directories needs a `FileSystem` and a `Path` implementation, provided once at the edge — from `@effect/platform-node` on Node.

## Your first task: give your CLI a home

Say you are building a CLI called `todo`, and you want it to keep its settings in the right place on every operating system, creating the directory the first time it writes there. Here is that goal, start to finish.

### 1. Build the AppDirs layer for your namespace

`AppDirs.layer` takes your application's namespace and resolves the config, data, cache and state directories for it. It is a layer-returning *function*, not a layer — calling it twice builds two independent services — so bind its result to a `const` and provide that one binding everywhere:

```ts
import { AppDirs } from "@effected/xdg";

const AppDirsLive = AppDirs.layer({ namespace: "todo", native: true });
```

`native: true` opts in to the OS-native directories on macOS and Windows; leave it off and the resolution stays pure XDG on every platform. More on that below.

### 2. Provide the environment and the platform

`AppDirs` resolves against two things: the `Xdg` environment (`$HOME` and the XDG variables, read once at layer construction) and a `FileSystem` / `Path` implementation for creating directories. Wire `Xdg.layer` and the Node platform layers underneath it:

```ts
import { AppDirs, Xdg } from "@effected/xdg";
import { NodeFileSystem, NodePath } from "@effect/platform-node";
import { Layer } from "effect";

const PlatformLive = Layer.mergeAll(NodeFileSystem.layer, NodePath.layer);

const AppDirsLive = AppDirs.layer({ namespace: "todo", native: true }).pipe(
  Layer.provide(Layer.mergeAll(Xdg.layer, PlatformLive)),
);
```

### 3. Read the paths, then create one on disk

Reading a directory is a plain property access — `app.dirs.config` is a `string`, not an `Effect`, because the environment was already resolved when the layer was built. The only members that touch the filesystem, and so the only ones that can fail, are the `ensure*` operations:

```ts
import { AppDirs } from "@effected/xdg";
import { Effect } from "effect";

const program = Effect.gen(function* () {
  const app = yield* AppDirs;

  // A value, not an Effect — reading a path cannot fail.
  const configDir = app.dirs.config;

  // Fallible: this actually creates the directory (mkdir -p) and returns it.
  const created = yield* app.ensureConfig;

  return { configDir, created };
});
```

### 4. Run it

Provide the layer, run the program, and you have a validated home for your CLI's config — created only when you asked for it:

```ts
import { AppDirs, Xdg } from "@effected/xdg";
import { NodeFileSystem, NodePath } from "@effect/platform-node";
import { Effect, Layer } from "effect";

const PlatformLive = Layer.mergeAll(NodeFileSystem.layer, NodePath.layer);

const AppDirsLive = AppDirs.layer({ namespace: "todo", native: true }).pipe(
  Layer.provide(Layer.mergeAll(Xdg.layer, PlatformLive)),
);

const program = Effect.gen(function* () {
  const app = yield* AppDirs;
  const configDir = app.dirs.config;
  const created = yield* app.ensureConfig;
  return { configDir, created };
});

Effect.runPromise(program.pipe(Effect.provide(AppDirsLive))).then(console.log);
// Linux, XDG_CONFIG_HOME set:   { configDir: "$XDG_CONFIG_HOME/todo", created: "$XDG_CONFIG_HOME/todo" }
// Linux, XDG_CONFIG_HOME unset: { configDir: "$HOME/.todo",           created: "$HOME/.todo" }
// macOS, native: true:         "$HOME/Library/Application Support/todo"
```

That is the whole workflow: build a layer for your namespace, ask for the service, read a path or create one. Everything below is a variation on it.

## Common operations

Each snippet reuses the `AppDirsLive` layer from above and runs inside the same `Effect.gen` / `Effect.provide` harness.

### Read every directory kind

`app.dirs` is the resolved record. `config`, `data`, `cache` and `state` are always present strings; `runtime` is present only when `$XDG_RUNTIME_DIR` is set (or you override it), so it is typed `string | undefined`. `configSearchPath` and `dataSearchPath` list where to *look* for files, in priority order:

```ts
const program = Effect.gen(function* () {
  const app = yield* AppDirs;
  return {
    config: app.dirs.config,
    data: app.dirs.data,
    cache: app.dirs.cache,
    state: app.dirs.state,
    runtime: app.dirs.runtime, // string | undefined — absent unless $XDG_RUNTIME_DIR is set
    configSearchPath: app.dirs.configSearchPath, // ["<app config>", "/etc/xdg/todo", …]
  };
});
```

The runtime directory has no fallback: a runtime directory must be user-owned and mode 0700, so when `$XDG_RUNTIME_DIR` is unset the key is simply absent rather than pointing somewhere invented.

### Create every directory at once

`ensure` creates each resolved directory that exists — config, data, cache, state, and runtime if there is one — and returns the whole resolved record. Use it at startup when you want every directory present before your program does any work:

```ts
const program = Effect.gen(function* () {
  const app = yield* AppDirs;
  const dirs = yield* app.ensure; // mkdir -p every directory, then return them all
  return dirs.cache;
});
```

For finer control, the per-kind `ensureConfig`, `ensureData`, `ensureCache` and `ensureState` each create and return one directory, and `ensureRuntime` returns an `Option<string>` — `none` when there is no runtime directory, so nothing is created and nothing fails.

### Use the OS-native directories

Pass `native: true` and, on macOS and Windows, resolution prefers the platform's own convention over the bare-XDG fallback — `~/Library/Application Support/todo` on macOS, `%APPDATA%\todo` on Windows. On Linux there is nothing to override, because XDG *is* the native convention there, so the flag changes nothing:

```ts
const AppDirsNative = AppDirs.layer({ namespace: "todo", native: true }).pipe(
  Layer.provide(Layer.mergeAll(Xdg.layer, PlatformLive)),
);
```

`native` is consulted only when no XDG environment variable and no explicit override already answered — it is one rung of a precedence ladder, not a switch that overrides everything.

### Override a specific directory

`dirs` lets you pin an absolute path for one or more kinds, and an override wins outright over every other rung. This is how you honor a `--cache-dir` flag or an environment variable your app reads itself, without giving up XDG resolution for the other kinds:

```ts
const AppDirsCustom = AppDirs.layer({
  namespace: "todo",
  dirs: { cache: "/var/tmp/todo-cache" }, // absolute; wins over XDG for cache only
}).pipe(Layer.provide(Layer.mergeAll(Xdg.layer, PlatformLive)));
```

### Load a config file from the XDG search path

`XdgConfig` is the bridge into [`@effected/config-file`](/config-file): `resolver` searches the app's whole config search path, `nativeResolver` probes the OS-native directory, and `savePath` names the default write target. Drop them into a `ConfigFile` layer and the loader searches `~/.config/todo/config.json`, then each `$XDG_CONFIG_DIRS` entry, then the native directory — first match wins:

```ts
import { ConfigFile, JsonCodec, MergeStrategy } from "@effected/config-file";
import { AppDirs, Xdg, XdgConfig } from "@effected/xdg";
import { NodeFileSystem, NodePath } from "@effect/platform-node";
import { Layer, Schema } from "effect";

class TodoSettings extends Schema.Class<TodoSettings>("TodoSettings")({
  editor: Schema.String,
  color: Schema.Boolean,
}) {}

class TodoConfig extends ConfigFile.Service<TodoConfig, TodoSettings>()("todo/Config") {}

const AppDirsLive = AppDirs.layer({ namespace: "todo", native: true }).pipe(
  Layer.provide(Layer.mergeAll(Xdg.layer, NodeFileSystem.layer, NodePath.layer)),
);

export const TodoConfigLive = ConfigFile.layer(TodoConfig, {
  schema: TodoSettings,
  codec: JsonCodec,
  resolvers: [
    XdgConfig.resolver({ filename: "config.json" }),
    XdgConfig.nativeResolver({ namespace: "todo", filename: "config.json" }),
  ],
  strategy: MergeStrategy.firstMatch<TodoSettings>(),
  defaultPath: XdgConfig.savePath("config.json"),
}).pipe(Layer.provide(AppDirsLive));
```

Order matters: put `resolver` before `nativeResolver` so an existing `~/.config/todo/config.json` still wins over the OS-native directory. `savePath` names where `ConfigFile.save` writes by default; it does not create the directory, because `save` already creates the parent of whatever path it is handed.

## Handling failures

There are exactly two typed errors, because path resolution is not IO — only creating a directory is. Both are tagged errors carrying their cause structurally, so you route on the tag with `Effect.catchTag`:

- **`XdgEnvError`** — the environment cannot satisfy resolution. It is raised for one thing only: `$HOME` is not set. Every other XDG variable is optional by construction, so its absence is a resolved default, not a failure. Because resolution happens when the layer is built, this surfaces the moment you provide the layer, before any `AppDirs` exists.
- **`AppDirsError`** — a directory could not be created. It carries `directory` (which kind), `path` (where), and the structural `cause`. It is the *only* way `AppDirs` fails, because resolution already succeeded; reaching it means the `mkdir -p` itself did not work.

```ts
import { AppDirs, AppDirsError, Xdg, XdgEnvError } from "@effected/xdg";
import { NodeFileSystem, NodePath } from "@effect/platform-node";
import { Effect, Layer } from "effect";

const AppDirsLive = AppDirs.layer({ namespace: "todo", native: true }).pipe(
  Layer.provide(Layer.mergeAll(Xdg.layer, NodeFileSystem.layer, NodePath.layer)),
);

const program = Effect.gen(function* () {
  const app = yield* AppDirs;
  return yield* app.ensureConfig;
}).pipe(
  Effect.provide(AppDirsLive),
  // The environment was unusable: $HOME is unset.
  Effect.catchTag("XdgEnvError", (error: XdgEnvError) => Effect.succeed(`env error: ${error.variable} is not set`)),
  // The directory could not be created — report which kind and where.
  Effect.catchTag("AppDirsError", (error: AppDirsError) =>
    Effect.succeed(`could not create the ${error.directory} directory at ${error.path}`),
  ),
);

Effect.runPromise(program).then(console.log);
```

One case is deliberately *not* a typed error: a namespace that is empty, contains a path separator, or is exactly `.` or `..` is a **defect** at layer construction, not a failure in the error channel. It can only come from your own code — `namespace: "../.."` would resolve the app's directories outside `$HOME` entirely — so it dies loudly rather than routing quietly. Give `AppDirs.layer` a single, well-formed path component and it never happens.
