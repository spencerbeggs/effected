# @effected/ts-vfs

TypeScript virtual file systems for Effect. Fetch a package's declaration files from npm through the jsDelivr CDN, cache them on disk, resolve an import specifier against the manifest that shipped with them, and hand the result to `@typescript/vfs` as a real `VirtualTypeScriptEnvironment` — so Twoslash-style documentation tooling can typecheck a code sample that imports a library the machine has never installed.

## Why @effected/ts-vfs

A documentation site that typechecks its own examples needs the types of every package those examples import, and it needs them without a `node_modules` tree. That means fetching `.d.ts` files over the network, which means caching them, which means answering the two questions caching always asks: is this entry live, and if it is not, is it *missing* or merely *stale*?

The distinction is the package. A miss has to be fetched. A stale entry has files on disk already, so the build can either refresh them or — offline, or in CI without network — serve them as they are and keep working. Collapsing the two into a boolean makes the second behavior impossible to express, and the failure mode is a docs build that dies on a plane.

Everything else follows from that: a version reference resolves locally rather than through a CDN endpoint that may not exist, paths that arrive from a CDN are treated as untrusted data, and the TypeScript compiler is loaded only if you actually ask for a compiler.

## Install

```bash
npm install @effected/ts-vfs effect
```

```bash
pnpm add @effected/ts-vfs effect
```

Requires Node.js >=24.11.0. `dependencies` is empty: `effect` v4, [`@effected/store`](/store), [`@effected/xdg`](/xdg) and [`@effected/semver`](/semver) are peer dependencies, and `typescript` plus `@typescript/vfs` are **optional** peers. Install the optional pair only if you call `TsEnvironment.make` — every other module works without a compiler on disk.

The store edge is what makes this package integrated tier, and it is load-bearing: store's `Cache` appears in `TypeCache.layer`'s requirements, so a single copy of it in your graph is the point. It is a peer for exactly that reason.

## Quick start

`TypeRegistry` is the facade — one service over the cache, the fetcher and the resolver:

```ts
import { PackageSpec, TypeRegistry } from "@effected/ts-vfs";
import { Effect } from "effect";

const program = Effect.gen(function* () {
  const registry = yield* TypeRegistry;

  // A version reference — exact, dist-tag or range — pinned to one version.
  const version = yield* registry.resolveVersion("zod", "^3.23.0");

  // The declaration files, fetched and cached on first call, read from disk after.
  return yield* registry.getPackageVfs(PackageSpec.make({ name: "zod", version }));
  // Map<string, string>: "node_modules/zod/index.d.ts" → source
});
```

Wiring is composed at the edge, and every layer static is a **parameterized factory** — bind the built layer to a `const`, or two provide sites mint two caches:

```ts
import { PackageFetcher, TypeCache, TypeRegistry } from "@effected/ts-vfs";
import { Cache } from "@effected/store";
import { AppDirs, Xdg } from "@effected/xdg";
import { NodeContext, NodeHttpClient } from "@effect/platform-node";
import { Layer } from "effect";

// Bind it once. Calling the factories again builds a second, separate cache.
const RegistryLive = TypeRegistry.layer.pipe(
  Layer.provide(Layer.mergeAll(TypeCache.layerXdg(), PackageFetcher.layer)),
  Layer.provide(Cache.layerSqlite({ filename: cacheDbPath })),
  Layer.provide(AppDirs.layer({ name: "my-docs" })),
  Layer.provide(Layer.mergeAll(Xdg.layer, NodeHttpClient.layerUndici, NodeContext.layer)),
);
```

`TypeCache.layerXdg()` roots the file plane at `<XDG cache>/<namespace>/` and creates that directory, which also discharges store's constraint that a database's parent directory must exist before the SQLite layer is built. This package never builds the store layer itself — you compose `Cache.layerSqlite` (or `Cache.layerTest`, at `:memory:`) yourself.

For a fixed directory instead, `TypeCache.layer({ cacheDir })` takes an absolute path; a relative one is developer wiring and dies at layer construction.

## The two-plane cache

`TypeCache` keeps the fetched files on disk under `<cacheDir>/<name>/<version>/` and the per-package metadata in [`@effected/store`](/store)'s `Cache`, which brings TTL, evict-on-read expiry and bulk pruning with it. Two planes, and the seam between them is where the useful behavior lives:

* **live metadata → hit.** Serve from disk.
* **files on disk, no live metadata → stale.** Refetch when `autoFetch` is on (the default); serve the on-disk files as they are when it is off.
* **nothing → miss.** Fetch, or fail with a typed `PackageNotFoundError` under `autoFetch: false`.

That last column is what an offline build wants: `getPackageVfs(spec, { autoFetch: false })` never touches the network, serves whatever it has, and tells you — typed — when it has nothing.

Removal deletes metadata *before* files, so a crash mid-remove leaves harmless orphaned files rather than a phantom cache hit. Expiry rides store's clock, which means `TestClock` drives it and no test ever sleeps.

## Fetching, and paths as data

`PackageFetcher` is the jsDelivr client, and everything it returns is untrusted input. File paths from a CDN response are data, never path fragments to be trusted: `TypeCache` rejects absolute paths and `..` segments as typed `TypeCacheError`s before any join, and `PackageSpec`'s schema constrains name and version so a spec that could escape the cache directory is simply unconstructable.

The same posture runs through the resolver. Exports-map wildcard substitution builds its maps with `Object.create(null)` and skips dunder keys — the predecessor library happily assigned `__proto__` out of a fetched exports map. Recursion is depth-capped, wildcard patterns are bounded before they reach a regex, and `getTypeFiles` has a materialization budget, so a hostile package cannot exhaust memory or hang the build.

Failures are structured and specific: `FetchError` (with a `status` and a `kind` of `status`, `body`, `schema` or `transport`), `PackageNotFoundError`, `VersionNotFoundError`.

## Version resolution is local

`resolveVersion(name, ref)` takes a dist-tag, an exact version or a semver range and returns one pinned version string. Tags resolve through the CDN's tag map, exact versions match the published list, and ranges resolve **locally** with [`@effected/semver`](/semver) — max-satisfying over the published versions. There is no `/resolve` endpoint and no parsing of error prose to find out what went wrong. A reference that matches nothing fails as a typed `VersionNotFoundError` carrying the versions that were available.

## Resolving imports

`TypeResolver` is pure statics — no service, no layer. Through the registry, it answers the question a Twoslash sample actually asks: given `import { z } from "zod/v4"`, which `.d.ts` file is that?

```ts
import { PackageSpec, TypeRegistry } from "@effected/ts-vfs";
import { Effect, Option } from "effect";

const program = Effect.gen(function* () {
  const registry = yield* TypeRegistry;
  const zod = PackageSpec.fromString("zod@3.23.8");

  const resolved = yield* registry.resolveImport(zod, "zod/v4");
  // Option<ResolvedModule> — None when the manifest offers no evidence for the subpath

  const entries = yield* registry.getTypeEntries(zod);
  // every declared type entry point

  return Option.getOrUndefined(resolved) ?? entries[0];
});
```

`PackageSpec.normalizeSpecifier` turns an arbitrary import back into the package to fetch: `"node:fs"` and Node's built-ins become `"node"` (the `@types/node` convention), `"@effect/platform/Http"` becomes `"@effect/platform"`, `"lodash/fp"` becomes `"lodash"`.

## Batch loading

`getVfs` loads several packages concurrently and merges the results. It is best-effort by design: per-package failures accumulate, the partial VFS still merges, and it fails — with a structured `BatchLoadError` carrying every failure — only when *every* package failed. An empty array is not an error; it is an empty VFS.

```ts
import { PackageSpec, TypeRegistry } from "@effected/ts-vfs";
import { Effect } from "effect";

const program = Effect.gen(function* () {
  const registry = yield* TypeRegistry;
  return yield* registry.getVfs([
    PackageSpec.fromString("zod@3.23.8"),
    PackageSpec.fromString("effect@latest"),
  ]);
});
```

One package's 404 should not take a docs build down when the other nine resolved.

## Virtual packages

Not every declaration comes from npm. `VirtualPackage` builds a synthetic package out of declaration text you already have — API Extractor output, hand-written ambient declarations, the package the docs site is documenting — and drops it into the same VFS as the fetched ones. Instances are transient and never touch the disk cache.

```ts
import { mergeVfs, VirtualPackage } from "@effected/ts-vfs";

const local = VirtualPackage.create(
  "@my-org/api-types",
  "1.0.0",
  "export interface User { readonly id: string }",
);

const vfs = mergeVfs(remoteVfs, local.toVfs());
// node_modules/@my-org/api-types/{package.json,index.d.ts}
```

`createMultiEntry` takes a map of entry files and generates the `exports` map to match, so TypeScript's module resolution works against the synthetic manifest. `fromFile` reads a `.d.ts` from disk through Effect's `FileSystem`.

## Building a TypeScript environment

`TsEnvironment.make` is the `@typescript/vfs` seam: a `Vfs` plus compiler options in, a `VirtualTypeScriptEnvironment` out, ready for a language service.

```ts
import { TsEnvironment } from "@effected/ts-vfs";
import { Effect } from "effect";
import ts from "typescript";

const program = Effect.gen(function* () {
  const environment = yield* TsEnvironment.make({
    vfs,
    compilerOptions: { strict: true, target: ts.ScriptTarget.ES2022 },
  });

  environment.createFile("index.ts", `import { z } from "zod";`);
  return environment.languageService.getSemanticDiagnostics("index.ts");
});
```

This is the **only** module that touches the optional peers, and it loads them lazily inside `make` — a consumer that never calls it never loads the compiler, and a missing peer fails as a typed `TsEnvironmentError` rather than crashing at import time. `projectRoot` defaults to `process.cwd()` and is the root every relative VFS key is re-rooted under.

## Events

`RegistryObserver` is an opt-in event channel. Provide no observer and you pay nothing: every emission site resolves the service through `Effect.serviceOption` and no-ops on absence. The library never logs.

```ts
import { RegistryObserver } from "@effected/ts-vfs";

const ObserverLive = RegistryObserver.layerCallback((event) => {
  switch (event._tag) {
    case "CacheHit":
      return console.log(`${event.package} — cached`);
    case "PackageLoadFailed":
      return console.warn(`${event.package} failed: ${event.kind}`);
    default:
      return;
  }
});
```

`RegistryEvent` is a schema-backed tagged union — `VersionResolved`, `VersionResolveFailed`, `CacheHit`, `CacheStale`, `CacheMiss`, `FetchStart`, `FetchFailed`, `PackageLoaded`, `PackageLoadFailed`, `BatchStart`, `BatchComplete` — because events cross the library/host boundary and hosts ship them to telemetry. Narrow with `switch (event._tag)` or `Match`.

`PackageLoadFailed.kind` is computed from typed error tags and the structured `FetchError.status`, never from matching substrings in an error message. `RegistryObserver.layerNoop` is there when a test needs the service present and silent.
